# Детектор «оплата через инфраструктуру Avito» (признак `apay`)

> **Для соседней сессии, автоматизирующей парсинг.** Этот документ — полное
> правило определения, продаётся ли контрафактная карточка на Avito **с оплатой
> через инфраструктуру Avito** (Авито Доставка / безопасная сделка). Рабочий
> референс-скрипт: **`scraper/enrich_avito_pay.py`** (рядом с этим файлом).
>
> Краткая версия правила есть в `MONITORING_LOGIC.md` (раздел «💳 Признак apay»).

---

## 1. Зачем это нужно (юридический контекст)

Avito официально заявляет:
- «Мы не маркетплейс, а **доска объявлений**».
- «Несём ответственность только за объявления **с оплатой через нашу
  инфраструктуру**» (Авито Доставка, безопасная сделка, оплата на счёт Avito).

Значит: контрафактная карточка Microsoft Office, которую можно **купить через
Avito** (а не «написать продавцу и заплатить мимо») — это **прямая зона
ответственности Avito** по их собственной логике. Поле `apay=true` помечает
именно такие карточки.

Это сильный аргумент для обращения: «вот N карточек контрафакта, проданных через
вашу платёжную инфраструктуру — извольте отвечать».

---

## 2. Где искать признак

Признак виден **только на странице самого объявления**
(`https://www.avito.ru/.../{slug}_{cardId}`), НЕ в поисковой выдаче.
В выдаче бейдж «Доставка» появляется нестабильно и не равен возможности оплаты.

⚠️ Поэтому детекция требует **отдельного захода на каждую карточку** — это
медленно и Avito банит (см. § 5).

---

## 3. Правило детекции (что считать `apay=true`)

### ✅ Признак ОПЛАТЫ через Avito (apay=true)
На странице объявления есть **реальная кнопка покупки для покупателя**:
- `Купить с доставкой`
- `Заказать с доставкой`
- `Добавить в корзину`
- `Купить сейчас`
- `Оформить заказ`
- `Купить с Авито Доставкой`

JS-регекс (по `document.body.innerText`):
```js
/Купить с доставкой|Заказать с доставкой|Добавить в корзину|Купить сейчас|Оформить заказ|Купить с Авито Доставкой/i
```

### ❌ Признак оплаты ВНЕ Avito (apay=false)
В блоке действий только:
- `Написать`
- `Показать телефон`

— и **нет** кнопки покупки выше. Продавец принимает деньги напрямую, минуя Avito.

### ⚠️ ЛОЖНЫЙ признак — НЕ использовать!
Текст **«Отправите Авито Доставкой?»** есть почти на КАЖДОЙ карточке —
это маркетинговое приглашение **продавцу** оформить доставку, а НЕ признак
что объявление уже принимает оплату через Avito. Детектор его игнорирует.

Также `data-marker="safedeal-item-header"` присутствует на всех карточках,
но часто **пустой** (`innerText === ''`) — сам по себе не признак. Можно
использовать только как вторичный сигнал, когда блок непустой.

## 4. Точный код детектора

Из `scraper/enrich_avito_pay.py`, выполняется через `page.evaluate()` после
`goto` на страницу объявления + прокрутка:

```js
() => {
  const body = document.body.innerText || '';
  // Реальная кнопка покупки через Avito (для покупателя)
  const canBuy = /Купить с доставкой|Заказать с доставкой|Добавить в корзину|Купить сейчас|Оформить заказ|Купить с Авито Доставкой/i.test(body);
  // Блок безопасной сделки активен (не пустой) — вторичный сигнал
  const sd = document.querySelector('[data-marker="safedeal-item-header"]');
  const safedealActive = !!(sd && sd.innerText && sd.innerText.trim().length > 0);
  // Заблокировано / удалено
  const blocked = /Доступ ограничен|Объявление снято|больше не доступно/i.test(body.slice(0,300));
  return { canBuy, safedealActive, blocked };
}
```

`apay = canBuy` (основной признак). `safedealActive` — опциональная проверка.

## 5. Антибан (критично для автоматизации)

Avito банит после **~220 заходов на страницы товара подряд** с одного IP
(даже через CloakBrowser+VDSina). Признаки бана: `blocked === true`,
title «Доступ ограничен».

Меры в `enrich_avito_pay.py`:
- Пауза между карточками **2.5–5 сек** (`random.uniform(2.5, 5.0)`)
- При 5 блокировках подряд — **cooldown 60 сек**, потом продолжаем
- **Возобновление**: результат пишется в `avito_pay.json` каждые 20 карточек;
  при перезапуске карточки с `None` (блок/ошибка) перепроверяются, с `true/false`
  — пропускаются. Так можно гонять в несколько заходов.

Для **автоматического парсинга** рекомендуется:
- Детектить `apay` только для **новых** карточек (которых нет в БД), а не всех
- Растянуть во времени: 200 карточек за прогон, остальное — следующим запуском
- Или residential-proxy с ротацией IP (тогда лимит выше)

## 6. Формат данных

`avito_pay.json` (в `cloakbrowser-lab/`, в .gitignore — это сырой результат):
```json
{
  "AV-8090573777": true,
  "AV-8080683083": false,
  "AV-8165063692": null
}
```
- `true` — оплата через Avito (apay)
- `false` — оплата вне Avito
- `null` — не проверено (блок/ошибка) → перепроверить

В MON_RAW (дашборд) поле `apay:true` добавляется **только для true-карточек**
(чтобы не раздувать). Отсутствие = falsy. Подмешивается в
`import_cloak_full.py` через карту `AVITO_PAY`.

## 7. Для Supabase (соседняя сессия)

Добавить в таблицу `listings` колонку:
```sql
ALTER TABLE listings ADD COLUMN avito_pay BOOLEAN;  -- NULL = не проверено
```

Пайплайн автопарсинга:
1. Основной парсинг выдачи (как обычно) → upsert карточек
2. Отдельный enrichment-воркер: берёт `WHERE pl='avito' AND avito_pay IS NULL`,
   открывает страницы товара, проставляет `avito_pay`, ограничивая ~200/прогон
   с паузами 2.5-5 сек (см. § 5)
3. Дашборд `index_new.html`: фильтр `avito_pay = true`, бейдж 💳, колонка в экспорте

На клиенте дашборда логика уже есть в `anti-ms-dashboard/index.html`:
- `monState.apay` ('all' | 'yes')
- `monToggleApay()`, фильтр в `monApplyFilters()`
- бейдж в `monRender()` (`r.apay === true`)
- колонка «Оплата через Avito» в `exportMonCSV` / `exportMonXLSX`
